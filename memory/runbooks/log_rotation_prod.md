# Runbook: Apply log rotation to Shelfsort supervisor logs

**When to run:** Once, in your next prod maintenance window.
**Why:** `/var/log/supervisor/backend.{out,err}.log` and the matching
`frontend.*.log` files have no size cap by default. Logs grow
unboundedly and will eventually fill the disk. After this change,
each file is capped at 10MB × 5 rotations = 50MB per stream max.

**Risk:** Very low. Worst case the backend restarts (~5 s of
downtime) if `supervisorctl update` decides to bounce it. No data
loss possible — supervisor only manages process lifecycle, not app
state.

**Why this can't be done from the Emergent agent:** The preview
container's `/etc/supervisor/conf.d/supervisord.conf` is marked
`# READONLY FILE, DO NOT EDIT THIS FILE` and is regenerated on every
pod restart. The agent has no SSH access to your prod box.

---

## Steps (copy-paste ready)

### 1. SSH to your production server
```bash
ssh youruser@your-prod-host
```

### 2. Back up the current config (one-liner)
```bash
sudo cp /etc/supervisor/conf.d/backend.conf /etc/supervisor/conf.d/backend.conf.bak.$(date +%Y%m%d)
sudo cp /etc/supervisor/conf.d/frontend.conf /etc/supervisor/conf.d/frontend.conf.bak.$(date +%Y%m%d)
```

### 3. Edit the backend config
```bash
sudo nano /etc/supervisor/conf.d/backend.conf
```

Inside the `[program:backend]` block, **add these four lines** (place
them right under the existing `stdout_logfile` and `stderr_logfile`
lines):

```ini
stdout_logfile_maxbytes=10MB
stdout_logfile_backups=5
stderr_logfile_maxbytes=10MB
stderr_logfile_backups=5
```

Save and exit (Ctrl+O, Enter, Ctrl+X in nano).

### 4. Do the same for the frontend config
```bash
sudo nano /etc/supervisor/conf.d/frontend.conf
```

Same four lines inside the `[program:frontend]` block.

### 5. Reload supervisor so it picks up the change
```bash
sudo supervisorctl reread
sudo supervisorctl update
```

`reread` parses the configs without touching running processes.
`update` applies them — supervisor will only restart programs whose
config actually changed (it should restart `backend` and `frontend`).

### 6. Verify the programs came back up
```bash
sudo supervisorctl status
```

Expected output (timestamps will differ):
```
backend                          RUNNING   pid 12345, uptime 0:00:08
frontend                         RUNNING   pid 12346, uptime 0:00:08
mongodb                          RUNNING   pid 12347, uptime 5 days, ...
```

If anything shows `FATAL` or `BACKOFF`, immediately:
```bash
sudo cp /etc/supervisor/conf.d/backend.conf.bak.YYYYMMDD /etc/supervisor/conf.d/backend.conf
sudo cp /etc/supervisor/conf.d/frontend.conf.bak.YYYYMMDD /etc/supervisor/conf.d/frontend.conf
sudo supervisorctl reread && sudo supervisorctl update
```
to revert.

### 7. Sanity-check the live app
```bash
curl -s https://YOUR-PROD-DOMAIN/api/maintenance-banner
# Should return `null` or a JSON object — not connection refused.
```

Visit `https://YOUR-PROD-DOMAIN/changelog` in a browser. The canary
uptime pill and the GitHub Actions badge should both render.

### 8. (Optional) Confirm rotation is working
You won't see rotated files (`backend.out.log.1`, `.2`, etc.) until
the active log first crosses 10MB. To verify the config landed
without waiting:
```bash
sudo supervisorctl avail
sudo cat /etc/supervisor/conf.d/backend.conf | grep -E 'logfile_(maxbytes|backups)'
```
You should see all four lines you added.

---

## Done — what you've gained

- Each of the four log streams (backend stdout/stderr, frontend
  stdout/stderr) capped at **50MB max** (10MB × 5 rotations).
- Supervisor auto-rotates: when `backend.out.log` hits 10MB, it
  becomes `backend.out.log.1`, a fresh `backend.out.log` is created,
  and the oldest rotation (`.5`) is deleted.
- No more slow-motion disk-space leak from unbounded log growth.
- Roadmap reminder #14 can be marked ✅ DONE.
